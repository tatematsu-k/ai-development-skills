# バグ調査 (Bug Investigation)

## Overview

バグの内容を正確に把握し、再現テストを作成した上で、修正方針をユーザーと合意する。

**Announce at start:** 「バグ修正モードで開始します。まずバグの調査を行います。」

<IMPORTANT>
バグ修正モードでは要件定義・設計フェーズをスキップする。
ただし修正方針はユーザーの承認を得てから実装を開始すること。
</IMPORTANT>

## The Process

### Step 1: バグ内容のヒアリング

ユーザーとの対話で以下を明確化する:

1. **何が起きているか** — 現在の誤った動作を具体的に確認
2. **期待される動作** — 本来どう動くべきか
3. **再現手順** — どのような操作・条件で発生するか
4. **発生環境** — 環境依存の有無 (本番/ステージング/ローカル等)
5. **発生頻度** — 常に発生 / 特定条件のみ / 間欠的

### Step 2: 既存仕様の確認

バグの「あるべき姿」を判断するために、既存仕様を確認する:

1. **関連コードの調査** — 該当機能の実装を読み、現在のロジックを理解する
2. **既存テストの確認** — 該当機能のテストケースを確認し、想定された仕様を把握する
3. **ドキュメントの確認** — 仕様書・README等に記載がないか確認する

> もし「あるべき姿」がコード・テスト・ドキュメント間で矛盾する場合は、
> ユーザーに確認して正しい仕様を合意する。

### Step 3: バグ再現テストの作成

<IMPORTANT>
修正コードを書く前に、必ずバグを再現するテストを先に作成すること。
このテストは現時点で「失敗する」状態であること (Red)。
</IMPORTANT>

1. バグの再現条件をテストコードに落とし込む
2. テストを実行し、現在の実装で失敗することを確認する
3. 再現テストをコミットする

```bash
git add <test-files>
git commit -m "test: add failing test to reproduce <bug-description>

Co-Authored-By: Claude <noreply@anthropic.com>"
```

### Step 4: 修正方針の策定

バグの原因と修正アプローチを検討する:

1. **原因の特定** — なぜバグが発生しているかの根本原因を明らかにする
2. **修正アプローチの検討** — 以下を考慮して修正案を策定する:
   - 最小限の変更で修正できるか
   - 既存機能への副作用はないか
   - 同種のバグが他にも潜在していないか
3. **修正案の提示** — 修正方針をユーザーに説明し承認を得る

> 「**バグ調査結果:**
>
> - 原因: [根本原因の説明]
> - 修正方針: [修正アプローチ]
> - 影響範囲: [変更が及ぶ範囲]
> - リスク: [副作用の可能性]
>
> この方針で修正を進めてよいですか？」

<HARD-GATE>
修正方針をユーザーが承認するまで実装に進んではならない。
</HARD-GATE>

### Step 5: バグ調査レポートの作成

以下の構造でドキュメントを作成:

```markdown

## バグ概要
- 現象:
- 期待される動作:
- 再現手順:
- 発生頻度:

## 原因分析
- 根本原因:
- 該当コード:

## 既存仕様との整合性
- 既存テスト:
- ドキュメント:
- 合意した「あるべき姿」:

## 修正方針
- アプローチ:
- 変更対象ファイル:
- 影響範囲:
- リスク:

## 再現テスト
- テストファイル:
- テストケース:
```

### Step 6: 保存 & コミット

```bash
git add aidlc-docs/bugs/YYYY-MM-DD-<bug-name>.md
git commit -m "docs: add bug investigation report for <bug-name>

Co-Authored-By: Claude <noreply@anthropic.com>"
```

### Step 7: 次フェーズへ遷移

> 「バグ調査が完了しました。実装に進みます。」

**REQUIRED NEXT SKILL:** `aidlc:implementation`

> **注:** バグ修正が小規模 (1PR以内) の場合、サブタスク分割はスキップして直接実装に入る。
> 複数PRにまたがる場合は `aidlc:subtask-decomposition` を先に実行する。
