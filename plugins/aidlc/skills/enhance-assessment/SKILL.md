# エンハンス対象評価 (Enhance Assessment)

## Overview

既存機能の改善・拡張を行う前に、対象機能の特定・影響分析・後方互換性の確認を行う。

**Announce at start:** 「エンハンスモードで開始します。まず対象機能の評価を行います。」

## The Process

### Step 1: コンテキスト把握

既存コードベースを確認し、全体像を把握する:

- プロジェクト構成・技術スタック
- 既存機能の一覧と依存関係
- テストカバレッジの状況

### Step 2: エンハンス対象の特定

ユーザーとの対話で以下を明確化する:

1. **どの機能をエンハンスするか** — 対象機能を具体的に特定する
2. **何を改善したいか** — 機能追加 / パフォーマンス改善 / UX改善 / リファクタリング等
3. **なぜ改善が必要か** — 背景・動機・解決したい課題
4. **期待する成果** — エンハンス後のあるべき姿

### Step 3: 既存機能への影響分析

対象機能に関連するコードを調査し、影響範囲を明確化する:

1. **直接的な影響範囲**
   - 変更対象のファイル・モジュール・関数を特定
   - 変更対象が利用しているAPIやインターフェースを確認

2. **間接的な影響範囲**
   - 対象機能を利用している他のモジュール・機能を特定
   - 共有コンポーネント・ユーティリティへの影響を確認

3. **影響範囲サマリをユーザーに提示**

### Step 4: 後方互換性の確認

<IMPORTANT>
後方互換性を破壊する変更がある場合、必ずユーザーに明示し承認を得ること。
</IMPORTANT>

以下の観点で後方互換性を評価する:

1. **API互換性** — 既存のAPIシグネチャ・レスポンス形式が変わるか
2. **データ互換性** — DB スキーマ・データ形式に破壊的変更があるか
3. **振る舞い互換性** — 既存ユーザーの操作フローに影響があるか
4. **設定互換性** — 既存の設定値・環境変数が無効になるか

評価結果を以下のいずれかに分類:

- **完全互換**: 既存の動作に影響なし
- **互換あり (移行パスあり)**: 破壊的変更があるが、移行手段を提供可能
- **互換なし (破壊的変更)**: 既存の動作が変わる → ユーザー承認必須

```markdown
- 概要:

## 影響分析

- ...

- ...

## 後方互換性

- 詳細:
- 移行手段 (該当する場合):

- ...
```
