# 要件定義 (Requirement Definition)

## Overview

ユーザーの「やりたいこと」を対話を通じて明確な要件に落とし込む。

**Announce at start:** 「要件定義フェーズを開始します。やりたいことを詳しく教えてください。」

## The Process

### Step 1: コンテキスト把握

1. 既存のコードベース・ドキュメントを確認する
2. ブラウンフィールド(既存プロジェクトへの追加)か、グリーンフィールド(新規)かを判定
3. ブラウンフィールドの場合、関連するコードの構造を把握する

### Step 2: 要件の深掘り

ユーザーと1問ずつ対話して要件を明確にする。以下の観点を網羅する:

- **What**: 何を実現したいか
- **Why**: なぜそれが必要か (ビジネス上の理由)
- **Who**: 誰が使うか (ユーザー・システム)
- **Scope**: スコープ (何を含み、何を含まないか)
- **Constraints**: 制約条件 (技術的・ビジネス的)
- **Success Criteria**: 完了条件

**質問のルール:**
- 1メッセージにつき1問
- 可能な限り選択式で提示
- 推測できるものは提案して確認を取る

### Step 3: 要件定義書の作成

以下のテンプレートで作成する:

```markdown
# [機能名] 要件定義書

## 1. 概要
- 何を実現するか (1-2文)

## 2. 背景・目的
- なぜ必要か
- ビジネス上の期待効果

## 3. スコープ
### 含むもの
- ...

### 含まないもの
- ...

## 4. 機能要件
### FR-1: [要件名]
- 説明: ...
- 受け入れ条件: ...

## 5. 非機能要件
- パフォーマンス: ...
- セキュリティ: ...
- その他: ...

## 6. 制約条件
- 技術的制約: ...
- ビジネス制約: ...

## 7. 用語定義
| 用語 | 定義 |
|------|------|
| ... | ... |
```

### Step 4: 要件定義書の検証ループ (Evaluator-Driven)

作成した要件定義書の複雑性を判定し、必要に応じてエージェントによるレビューを実施する。
**最大2周**でループを終了する。

```
┌─────────────────────────────────────────────────┐
│         要件定義 検証ループ                       │
│                                                  │
│  複雑性判定 → エージェント選定 → Check → Act     │
│       ▲                                  │      │
│       └──────────────────────────────────┘      │
│                                                  │
│  終了条件: 全Evaluator PASS or 2周到達           │
│  ※ 軽微な要件はセルフチェックのみで通過          │
└─────────────────────────────────────────────────┘
```

#### Phase 1: Plan — 複雑性判定とエージェント選定

要件の複雑性を以下の基準で判定し、**必要なエージェントだけ**を立ち上げる。

**複雑性の判定基準:**

| レベル | 条件例 | 対応 |
|--------|-------|------|
| **S (軽微)** | 単一コンポーネントの小さな変更、バグ修正、文言変更 | エージェント不要。セルフチェックのみで Step 5 へ |
| **M (中規模)** | 1-2コンポーネントにまたがる機能追加、既存APIの拡張 | コアエージェントのみ |
| **L (大規模)** | 複数コンポーネント・複数デプロイ単位にまたがる、新規ドメイン概念の導入、外部連携 | フルレビュー |

**レベル別エージェント選定:**

| エージェント | S | M | L | 検証観点 |
|------------|---|---|---|---------|
| セルフチェック (エージェントなし) | ✅ | ✅ | ✅ | テンプレート準拠・明らかな漏れ |
| `ecc:doc-architect` (sonnet) | - | ✅ | ✅ | ドキュメント構造・Mermaid図・テンプレート充足度 |
| `ecc:architect` (opus) | - | ✅ | ✅ | 技術的実現可能性・アーキテクチャ影響 |
| `ecc:security-reviewer` (sonnet) | - | ※ | ✅ | セキュリティ要件の網羅性 |
| `aidlc:domain-expert` (opus) | - | - | ✅ | ドメイン妥当性・ユースケース網羅性・業務ルール |
| `aidlc:holistic-reviewer` (opus) | - | - | ✅ | 全体の一貫性・スコープ健全性・暗黙の前提 |
| `ecc:planner` (opus) | - | - | ✅ | 実装計画の実現可能性・フェーズ分割・リスク |

※ セキュリティ関連の要件 (認証/認可/データ保護/外部連携) を含む場合は M でも実行

> 「要件の複雑性: [S/M/L] と判定しました。
> [S の場合] セルフチェックのみで進めます。
> [M/L の場合] 検証ループを開始します (ラウンド 1/2)。有効なEvaluator: [一覧]」

**S の場合はセルフチェック (テンプレート準拠・必須セクションの充足) のみ行い、Step 5 に進む。**

#### Phase 2: Do — 要件定義書の整備

Step 3 で作成した要件定義書を確定する。
初回はそのまま Check フェーズへ、2周目以降は Act の修正を反映済みの状態で進む。

#### Phase 3: Check — レビュー実行

Phase 1 で選定したエージェントを**並列**で実行する。

各エージェントの検証観点:

**`ecc:doc-architect`** (sonnet)
- テンプレート準拠・各セクション充足度・曖昧な記述の検出

**`ecc:architect`** (opus)
- 技術的実現可能性・既存アーキテクチャとの整合性・影響範囲

**`ecc:security-reviewer`** (sonnet)
- セキュリティ関連の非機能要件の網羅性・認証/認可/データ保護

**`aidlc:domain-expert`** (opus)
- ドメインモデルの妥当性・ユースケース網羅性・業務ルールの明文化

**`aidlc:holistic-reviewer`** (opus)
- 要件間の整合性・スコープの健全性・暗黙の前提条件の可視化

**`ecc:planner`** (opus)
- 実装の実現可能性・フェーズ分割の妥当性・リスクと優先順位

```
REQUIREMENT REVIEW REPORT (Round N/2)
=====================================
[選定されたエージェントのみ表示]
doc-architect:      [PASS/WARN] (構造: OK, 充足度: X/Y セクション)
architect:          [PASS/WARN] (技術的妥当性: OK, 懸念: N件)
...

Verdict: [ALL PASS → Step 5 へ / IMPROVEMENTS FOUND → Act フェーズへ]
```

#### Phase 4: Act — 要件定義書の改善

evaluatorの指摘に基づき要件定義書を修正する:

| 指摘の種類 | 対応 |
|-----------|------|
| テンプレート未充足 | 不足セクションを自動補完 |
| 曖昧な記述 | より具体的な表現に修正 |
| セキュリティ要件の欠落 | 非機能要件セクションに追記 |
| 技術的実現性の懸念 | 制約条件セクションに追記 |
| ユースケースの漏れ (domain-expert指摘) | 機能要件セクションに追記 |
| 業務ルールの暗黙知 (domain-expert指摘) | 制約条件または機能要件に明文化 |
| フェーズ分割の改善 (planner指摘) | スコープセクションを調整 |
| スコープの問題 (holistic-reviewer指摘) | ⚠️ ユーザーに確認してから修正 |
| 要件の矛盾 | ⚠️ ユーザーに確認してから修正 |
| ドメインモデルの根本的な問題 | ⚠️ ユーザーに確認してから修正 |

**スコープや要件の本質に関わる変更は自動修正せず、必ずユーザーに確認する。**

修正後、Plan フェーズに戻り再検証 (ラウンド +1)。

> 「要件定義書の検証ループ完了 (ラウンド N/2)
> - 自動改善: X件 適用済み
> - ユーザー確認待ち: Y件 (ある場合)」

### Step 5: 保存とコミット

1. `aidlc-docs/requirements/YYYY-MM-DD-<feature>.md` に保存
2. gitにコミット

### Step 6: 次フェーズへの遷移

> 「要件定義書を作成しました: `<path>`
> 検証結果: 全evaluator PASS (ラウンド N で完了)
> 次はシステム設計フェーズに進みます。」

**REQUIRED NEXT SKILL:** `aidlc:system-architecture`
