# 要件定義 (Requirement Definition)

## Overview

ユーザーの「やりたいこと」を対話を通じて明確な要件に落とし込む。

**Announce at start:** 「要件定義フェーズを開始します。やりたいことを詳しく教えてください。」

## The Process

### Step 1: コンテキスト把握

1. 既存のコードベース・ドキュメントを確認する
2. ブラウンフィールド(既存プロジェクトへの追加)か、グリーンフィールド(新規)かを判定
3. ブラウンフィールドの場合、関連するコードの構造を把握する

### Step 2: 要件の深掘り (4サブフェーズ)

以下の4サブフェーズを順に進める。各サブフェーズはユーザーの指示でスキップ可能。
サブフェーズ開始時に「スキップする場合はその旨お伝えください」と案内する。

**質問のルール (全サブフェーズ共通):**
- 1メッセージにつき1問
- 可能な限り選択式で提示
- 推測できるものは提案して確認を取る

---

#### Phase 2-1: ヒアリング & 背景確認

作りたいものと、その背景にある課題を明確にする。

1. **What**: 何を実現したいか — ユーザーの要望を具体化する
2. **Why**: 解決したい問題は何か — 背景にある課題・動機を深掘りする
3. **Who**: 誰が使うか (ユーザー・システム)
4. **Success Criteria**: どうなれば成功か

---

#### Phase 2-2: 既存機能での実現可能性検討

新規開発の前に、既存の仕組みで解決できないかを検討する。

1. コードベース・既存機能・設定を調査し、要望を満たせる既存手段がないか確認する
2. 既存機能で実現可能な場合 → その方法を提案し、ユーザーの判断を仰ぐ
3. 既存機能では不十分な場合 → 不足点を明示し、次のフェーズへ進む

---

#### Phase 2-3: 機能妥当性 & 代替プラン検討

提案する機能の妥当性を検証する。別プランの方が明確に優れていると判断できる根拠がある場合のみ、比較検討を行う。

1. ユーザーが望む機能アプローチの妥当性を評価する
2. **代替プランが明確に優位と判断できる場合のみ**、複数プランを比較提案する
   - 各プランのメリット・デメリットを簡潔にまとめる (箇条書き)
   - 推奨プランとその理由を明示する
3. 妥当性に問題がなく代替プランの優位性もない場合 → そのまま次のフェーズへ進む

---

#### Phase 2-4: 制約確認 & 機能確定

制約事項を洗い出し、最終的な機能スコープを確定する。

1. **技術的制約**: 技術スタック・パフォーマンス・互換性など
2. **ビジネス制約**: スケジュール・リソース・外部依存など
3. **Scope**: 含むもの / 含まないもの を明確にする
4. 上記を踏まえ、最終的な機能仕様をユーザーと合意する

### Step 3: 要件定義書の作成

以下のテンプレートで作成する:

```markdown
# [機能名] 要件定義書

## 1. 概要
- 何を実現するか (1-2文)

## 2. 背景・目的
- なぜ必要か
- ビジネス上の期待効果

## 3. スコープ
### 含むもの
- ...

### 含まないもの
- ...

## 4. 機能要件
### FR-1: [要件名]
- 説明: ...
- 受け入れ条件: ...

## 5. 非機能要件
- パフォーマンス: ...
- セキュリティ: ...
- その他: ...

## 6. 制約条件
- 技術的制約: ...
- ビジネス制約: ...

## 7. 用語定義
| 用語 | 定義 |
|------|------|
| ... | ... |
```

### Step 4: 保存とコミット

1. `aidlc-docs/requirements/YYYY-MM-DD-<feature>.md` に保存
2. gitにコミット

### Step 5: 次フェーズへの遷移

> 「要件定義書を作成しました: `<path>`
> 次はシステム設計フェーズに進みます。」
