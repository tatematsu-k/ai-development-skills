# 実装・テスト (Implementation)

## Overview

サブタスク (=PR単位) の実装を行う。TDDベースで進め、意味のある単位でコミットする。

**Announce at start:** 「サブタスク #N の実装を開始します: [サブタスク名]」

**REQUIRED SUB-SKILL:** `aidlc:pr-strategy` — コミット・PRルールに従うこと

## The Process

### Step 1: ブランチ作成

```bash
git checkout -b feature/<feature-name>/<subtask-name>
```

命名規則: `feature/<機能名>/<サブタスク名>`

### Step 2: 実装 (TDDサイクル)

各機能単位で以下のサイクルを回す:

1. **Red**: 失敗するテストを書く
2. **Green**: テストを通す最小限のコードを書く
3. **Refactor**: コードを整理する
4. **Review**: 過剰実装がないか再評価する (下記チェックリスト参照)
5. **Commit**: 意味のある単位でコミット

#### 過剰実装チェックリスト (Refactor/Review時に確認)

- [ ] 今回のサブタスクのスコープ外の変更が混入していないか
- [ ] 1回しか使わないコードに不要な抽象化をしていないか
- [ ] 仮想的な将来要件のためのコードを書いていないか (YAGNI)
- [ ] 同じことをより少ないコードで実現できないか
- [ ] テストコード自体が過剰に複雑になっていないか

### Step 3: コミットルール

#### 通常のコミット

意味のある変更ごとにコミットする:

```bash
git add <specific-files>
git commit -m "feat: add order status validation

Co-Authored-By: Claude <noreply@anthropic.com>"
```

#### スクリプト生成ファイルのコミット

<IMPORTANT>
スクリプトによって生成・更新されるファイルは必ず別コミットにし、
`[scripts]` プレフィックスと実行したスクリプトをコミットメッセージに記載する。
</IMPORTANT>

```bash
# 1. スクリプトを実行
bin/rails generate migration AddStatusToOrders status:integer

# 2. 生成ファイルのみをコミット
git add db/migrate/YYYYMMDD_add_status_to_orders.rb
git commit -m "[scripts] rails generate migration AddStatusToOrders status:integer

Generated by: bin/rails generate migration AddStatusToOrders status:integer

Co-Authored-By: Claude <noreply@anthropic.com>"

# 3. 手動で編集した内容は別コミット
git add db/migrate/YYYYMMDD_add_status_to_orders.rb
git commit -m "feat: add status column with default value and index

Co-Authored-By: Claude <noreply@anthropic.com>"
```

### Step 4: PRサイズ監視

実装中に変更行数を定期的に確認:

```bash
git diff --stat HEAD~N  # またはブランチ分岐元との比較
```

**±150行を超えそうな場合:**
1. ユーザーに報告する
2. さらなる分割を提案する
3. または理由を説明して続行の承認を得る

### Step 5: ドキュメント更新

実装に伴い、関連ドキュメントを更新する:

- README やAPIドキュメントに影響がある場合は同期的に更新する
- 設定値・環境変数の追加があれば `.env.example` 等に反映する
- 既存ドキュメントとの整合性を確認する

<IMPORTANT>
ドキュメント更新は実装と同じPR内で行い、後回しにしない。
</IMPORTANT>

### Step 6: 実装完了確認

サブタスクの実装が完了したら:

1. 全テストが通ることを確認
2. 変更内容のサマリを作成
3. PR作成フェーズへ遷移

> 「サブタスク #N の実装が完了しました。
> - 変更ファイル数: X
> - 追加/削除行数: +Y/-Z
> - テスト結果: ALL PASS
>
> PR作成に進みます。」

**REQUIRED NEXT SKILL:** `aidlc:pr-creation`

## エラー・ブロッカー対応

**テストが失敗した場合:**
- 原因を調査し修正する
- 修正が困難な場合はユーザーに相談する
- **修正後の過剰実装チェック (必須):**
  - テスト修正のために追加したコードが本来のスコープを超えていないか確認する
  - 「テストを通すためだけ」の不要な抽象化・ヘルパー・防御的コードが混入していないか見直す
  - YAGNI原則に立ち返り、現時点で不要な汎用化・将来対応コードを除去する
  - 修正前後で変更行数が大幅に増加した場合は、より簡潔な別アプローチを検討する

**サイズ超過の場合:**
- ユーザーに分割案を提示する
- 承認を得てから対応する

**依存タスクの未完了:**
- 依存タスクが未完了の場合はスキップし、次の独立タスクに移る
- ユーザーに報告する
