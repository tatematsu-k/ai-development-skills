# リポジトリオンボーディング (Onboarding)

## Overview

新しいリポジトリ、または既存だがドキュメントが不足しているリポジトリに対して、コードベースの自動解析とユーザーとの対話を組み合わせ、開発に必要な基礎ドキュメント群を生成する。

**Announce at start:** 「オンボーディングを開始します。まずリポジトリを解析して全体像を把握します。」

## ドキュメント構成ルール

### 階層構造によるコンテキスト管理

AIのコンテキストウィンドウを効率的に使うため、ドキュメントは **概要ファイル + 詳細ディレクトリ** の2層構造で管理する。

```
$OUTPUT_DIR/
├── service-overview.md              # 概要 (サマリー + 詳細へのリンク)
├── service-overview/
│   ├── business-context.md          # ビジネス背景の詳細
│   └── related-services.md          # 関連サービスの詳細
├── features.md                      # 概要 (機能一覧 + 詳細へのリンク)
├── features/
│   ├── <feature-name-1>.md          # 各機能の詳細
│   └── <feature-name-2>.md
├── architecture.md                  # 概要 (構成図 + 詳細へのリンク)
├── architecture/
│   ├── directory-structure.md        # ディレクトリ構成の詳細
│   ├── data-flow.md                 # データフローの詳細
│   └── deployment.md                # デプロイ構成の詳細
├── development-guide.md             # 概要 (クイックスタート + 詳細へのリンク)
└── development-guide/
    ├── setup.md                     # 環境構築の詳細手順
    ├── testing.md                   # テストの詳細
    └── troubleshooting.md           # トラブルシューティング集
```

### セキュリティ情報の配置方針

セキュリティは独立ドキュメントにせず、関連する各ドキュメントのセクションとして記載する:

| セキュリティ観点 | 記載先 |
|----------------|--------|
| 認証・認可の方式 | `architecture.md` の「認証・認可」セクション |
| シークレット管理・環境変数 | `development-guide.md` の「シークレット管理」セクション |
| データ保護・暗号化 | `architecture.md` の「データ保護」セクション |
| セキュリティ上の注意事項 | `development-guide.md` の「セキュリティ注意事項」セクション |

### サイズガイドライン

- **概要ファイル**: 最大150行。サマリーと詳細ファイルへのリンクで構成
- **詳細ファイル**: 最大200行。1トピック=1ファイル
- 概要ファイルだけ読めばサービスの全体像が掴める状態にする
- 詳細は必要な時にだけ参照する想定

### リンク記法

概要ファイルから詳細ファイルへのリンクは以下の形式で統一する:

```markdown
> 詳細: [ディレクトリ構成の詳細](./architecture/directory-structure.md)
```

## The Process

### Step 1: リポジトリ自動解析

コードベースを広く探索し、以下の情報を収集する:

1. **プロジェクト構成**: ディレクトリ構造、主要ファイル、モノレポ構成の有無
2. **技術スタック**: 言語、フレームワーク、主要ライブラリ (package.json, requirements.txt, go.mod, Cargo.toml, Gemfile 等から判定)
3. **エントリポイント**: アプリケーションの起動点、ルーティング定義
4. **インフラ・デプロイ**: Dockerfile, docker-compose, CI/CD設定 (GitHub Actions, etc.), IaC (Terraform, CDK 等)
5. **テスト**: テストフレームワーク、テストディレクトリ、テスト実行コマンド
6. **既存ドキュメント**: README.md, CLAUDE.md, docs/, wiki 等
7. **セキュリティ関連**: 認証・認可の仕組み、環境変数管理、シークレット管理
8. **データストア**: DB (マイグレーション, スキーマ), キャッシュ, キュー等

解析後、把握した内容のサマリーをユーザーに提示する:

> 「リポジトリの解析が完了しました。以下が現時点での理解です:
>
> - **技術スタック**: ...
> - **構成**: ...
> - **主要コンポーネント**: ...
>
> いくつか確認したい点があります。」

### Step 2: ユーザーとの対話による補完

コード解析だけでは分からない情報を対話で収集する。以下の観点を網羅する:

- **サービスの目的**: このサービスは何を解決するのか、誰のためのものか
- **ドメイン知識**: 業界特有の用語や概念
- **運用状況**: 本番稼働中か、開発中か
- **チーム構成**: 関係するチームやステークホルダー
- **既知の技術的負債**: 把握している課題や改善予定
- **外部連携**: 外部API、サードパーティサービスとの連携

**質問のルール:**
- 1メッセージにつき1〜2問
- 可能な限り選択式で提示
- コード解析から推測できるものは提案して確認を取る
- ユーザーが「分からない」「スキップ」と答えた項目は省略してよい

### Step 3: 出力先の確認

ドキュメントの出力先をユーザーに確認する。デフォルトは `docs/` とする。

> 「ドキュメントの出力先はデフォルトで `docs/` を想定しています。変更しますか？」

ユーザーが承認した場合、または別のパスを指定した場合、そのパスを `$OUTPUT_DIR` とする。

### Step 4: ドキュメント生成

以下のドキュメントを順に生成する。各ドキュメントを生成するたびにユーザーに提示し、フィードバックがあれば反映する。

**生成順序:** 概要ファイル → ユーザー確認 → 詳細ファイル群 の順で進める。
情報量が少なくセクションが短い場合は、詳細ファイルを作らず概要ファイルに直接記載してよい。

#### 4-1: サービス概要

**概要ファイル** (`$OUTPUT_DIR/service-overview.md`):

```markdown
# [サービス名] サービス概要

## 1. サービスの目的
- このサービスは何か (1-2文)
- 解決する課題
- 対象ユーザー

## 2. 主要機能
| 機能 | 概要 |
|------|------|
| ... | ... |

## 3. 技術スタック
| カテゴリ | 技術 |
|----------|------|
| 言語 | ... |
| フレームワーク | ... |
| データベース | ... |
| インフラ | ... |
| CI/CD | ... |

## 4. システム構成図
(Mermaid等で記載)

## 5. 関連リポジトリ・サービス
| リポジトリ/サービス | 役割 |
|---------------------|------|
| ... | ... |

> 詳細: [ビジネス背景](./service-overview/business-context.md)
> 詳細: [関連サービスの詳細](./service-overview/related-services.md)
```

**詳細ディレクトリ** (`$OUTPUT_DIR/service-overview/`):
- `business-context.md` — ビジネス背景、市場、ステークホルダーの詳細
- `related-services.md` — 関連サービスとの連携詳細、API仕様等

#### 4-2: 機能概要

**概要ファイル** (`$OUTPUT_DIR/features.md`):

```markdown
# [サービス名] 機能概要

## 機能一覧

| 機能 | カテゴリ | 概要 | 詳細 |
|------|---------|------|------|
| [機能名] | [カテゴリ] | 1行説明 | [詳細](./features/<feature-name>.md) |
| ... | ... | ... | ... |

## データモデル概要
(主要エンティティとその関係をMermaid ER図で記載)
```

**詳細ディレクトリ** (`$OUTPUT_DIR/features/`):

各機能ごとに1ファイル:

```markdown
# [機能名]

## 概要
- ...

## エンドポイント/画面
- ...

## 主要ロジック
- ...

## 関連ファイル
- ...

## データモデル
(この機能に関連するモデルの詳細)
```

**概要ファイル** (`$OUTPUT_DIR/architecture.md`):

```markdown
# [サービス名] アーキテクチャ概要
```

**詳細ディレクトリ** (`$OUTPUT_DIR/architecture/`):
- `directory-structure.md` — 全ディレクトリの説明付きツリー
- `data-flow.md` — シーケンス図、データフロー図の詳細
- `deployment.md` — インフラ構成、CI/CDパイプラインの詳細

### Step 5: CLAUDE.md の生成・更新

既存の `CLAUDE.md` が存在するか確認する。

**既存の CLAUDE.md がある場合:**

> 「既存の CLAUDE.md が見つかりました。以下の選択肢があります:
> 1. 既存の内容を維持し、不足情報のみ追記する
> 2. 新しく生成した内容で置き換える
> 3. CLAUDE.md の更新をスキップする」

**CLAUDE.md が存在しない場合:**

以下のテンプレートで新規生成する:

```markdown
# [サービス名]

## プロジェクト概要
(1-2文でサービスの目的)

## 技術スタック
- 言語: ...
- フレームワーク: ...
- DB: ...

## ディレクトリ構成
(主要ディレクトリの役割)

## 開発コマンド
(頻出コマンド一覧)

## コーディング規約
(言語・フレームワーク固有のルール)

## テスト
(テスト実行コマンド、テストの書き方)

## 重要な注意事項
(このリポジトリ特有の注意点)
```

### Step 6: 保存とコミット

1. 生成した全ドキュメントを保存
2. git にコミット

```
docs: add onboarding documents

Generated: service-overview, features, architecture, development-guide
```

3. CLAUDE.md を更新した場合は別コミットにする

```
docs: add/update CLAUDE.md for AI-assisted development
```

### Step 7: 完了報告

> 「オンボーディングドキュメントの生成が完了しました:
>
> **概要ドキュメント** (全体像の把握に使用):
> - `$OUTPUT_DIR/service-overview.md`
> - `$OUTPUT_DIR/features.md`
> - `$OUTPUT_DIR/architecture.md` (認証・認可・データ保護を含む)
> - `$OUTPUT_DIR/development-guide.md` (シークレット管理・セキュリティ注意事項を含む)
>
> **詳細ドキュメント** (必要時に参照):
> - `$OUTPUT_DIR/<name>/` 配下に各トピックの詳細
>
> - `CLAUDE.md` — (生成/更新した場合)
>
> 概要ファイルだけで全体像が掴めます。
> 詳細は各概要ファイル内のリンクから辿ってください。」

## 補足ルール

- **既存ドキュメントの尊重**: 既に充実したドキュメントがある場合は、重複を避け差分のみ生成する
- **不明な情報**: コード解析でも対話でも判明しなかった項目は `<!-- TODO: 要確認 -->` コメントを残す
- **図の活用**: 構成図・フロー図は可能な限り Mermaid 記法で記載し、マークダウンで直接表示できるようにする
- **段階的な承認**: 各ドキュメントを生成するたびにユーザーに確認を取る。一括生成はしない
- **詳細ファイルの省略**: 情報量が少ないセクションは概要ファイルに直接記載し、詳細ファイルは作らない。空の詳細ファイルは作らないこと
