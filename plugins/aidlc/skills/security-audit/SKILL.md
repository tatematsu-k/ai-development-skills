# セキュリティ監査 (Security Audit)

## Overview

コードベースを OWASP Top 10 をベースとしたチェックリストで走査し、脆弱性を検出してリスク評価と修正提案を行う。

**Announce at start:** 「セキュリティ監査を開始します。まず対象範囲を確認させてください。」

## The Process

### Step 1: 監査スコープの確認

ユーザーに監査の対象範囲を確認する:

> 「監査の対象を確認します:
> 1. リポジトリ全体
> 2. 特定のディレクトリ/ファイル
> 3. 直近の変更 (git diff ベース)
>
> どの範囲を対象にしますか？」

また、以下の補足情報をユーザーに確認する:
- **重点観点**: 特に気になっている脆弱性カテゴリがあるか
- **除外対象**: テストコードやモックなど除外すべきファイルがあるか

### Step 2: 技術スタックの把握

対象コードベースの技術スタックを自動解析する:

- 言語・フレームワーク (検出パターンを決定するため)
- 依存ライブラリ (既知の脆弱性チェックのため)
- インフラ構成 (設定ミスの検出のため)

### Step 3: コード走査

以下のチェックリストに基づいてコードを走査する。検出された問題は一時的にリストに蓄積し、走査完了後にまとめて報告する。

#### チェックリスト

##### A01: アクセス制御の不備
- [ ] 認可チェックの欠落 (エンドポイント/APIに権限チェックがあるか)
- [ ] 水平権限昇格 (他ユーザーのリソースにアクセスできないか)
- [ ] 垂直権限昇格 (一般ユーザーが管理者機能にアクセスできないか)
- [ ] CORS の設定ミス
- [ ] ディレクトリトラバーサル

##### A02: 暗号化の失敗
- [ ] 平文でのパスワード/シークレット保存
- [ ] 弱い暗号アルゴリズムの使用 (MD5, SHA1 等)
- [ ] ハードコードされたシークレット/APIキー
- [ ] 不適切な乱数生成

##### A03: インジェクション
- [ ] SQL インジェクション (文字列結合によるクエリ構築)
- [ ] XSS (ユーザー入力の未エスケープ出力)
- [ ] コマンドインジェクション (外部入力を含むシェルコマンド実行)
- [ ] NoSQL インジェクション
- [ ] LDAP / XPath インジェクション

##### A04: 安全でない設計
- [ ] 入力バリデーションの欠落 (境界値、型、長さ)
- [ ] レート制限の欠如
- [ ] ビジネスロジックの欠陥

##### A05: セキュリティ設定のミス
- [ ] デバッグモードの本番有効化
- [ ] デフォルト認証情報の使用
- [ ] 不要なポート/サービスの公開
- [ ] セキュリティヘッダーの欠落
- [ ] 詳細なエラーメッセージの露出

##### A06: 脆弱なコンポーネント
- [ ] 既知の脆弱性を持つ依存ライブラリ
- [ ] メンテナンスされていないライブラリの使用
- [ ] ロックファイルの不整合

##### A07: 認証の失敗
- [ ] 弱いパスワードポリシー
- [ ] セッション管理の不備 (固定化、長すぎる有効期限)
- [ ] 多要素認証の欠如 (機密操作)
- [ ] ブルートフォース対策の欠如

##### A08: データの整合性の問題
- [ ] 安全でないデシリアライゼーション
- [ ] CI/CDパイプラインの整合性チェック欠如
- [ ] 署名なしの自動更新

##### A09: ログ・監視の不足
- [ ] セキュリティイベントのログ欠如 (ログイン失敗、権限エラー等)
- [ ] ログへの機密情報の記録
- [ ] 監視・アラートの欠如

##### A10: SSRF
- [ ] 外部入力によるURL/IPアドレスへのリクエスト送信
- [ ] 内部ネットワークへのアクセス制御の欠如

### Step 4: レポート生成

走査結果を以下の形式でレポートにまとめる。

**出力先**: ユーザーに確認する。デフォルトは `$OUTPUT_DIR/security-audit-YYYY-MM-DD.md`

```markdown
# セキュリティ監査レポート

**監査日**: YYYY-MM-DD
**対象**: [リポジトリ名 / 対象範囲]
**技術スタック**: [言語, フレームワーク]

## サマリー

| 深刻度 | 件数 |
|--------|------|
| Critical | N |
| High | N |
| Medium | N |
| Low | N |
| Info | N |

## 検出項目

### [CRITICAL/HIGH/MEDIUM/LOW/INFO] [項目タイトル]

- **カテゴリ**: A0X: [カテゴリ名]
- **深刻度**: Critical / High / Medium / Low / Info
- **該当箇所**: `path/to/file.py:42`
- **説明**: 何が問題か
- **リスク**: 悪用された場合の影響
- **修正提案**:
  ```diff
  - 問題のあるコード
  + 修正後のコード
  ```
- **参考**: [関連するセキュリティガイドラインへのリンク等]

(検出項目ごとに繰り返し)

## 未検査項目

以下の観点はこの監査では検査していない:
- ...

## 推奨アクション

優先度順の対応リスト:
1. [Critical] ...
2. [High] ...
3. ...
```

### Step 5: 深刻度の判定基準

| 深刻度 | 基準 |
|--------|------|
| Critical | 認証なしでリモートからデータ漏洩・システム制御が可能 |
| High | 認証済みユーザーによる権限昇格、機密データへのアクセス |
| Medium | 特定条件下で悪用可能、または情報漏洩のリスク |
| Low | セキュリティベストプラクティスからの逸脱、将来的なリスク |
| Info | 改善推奨だが直接的なセキュリティリスクは低い |

### Step 6: ユーザーへの報告と対応相談

レポートを生成したら、ユーザーに報告する:

> 「セキュリティ監査が完了しました。
>
> **検出結果**: Critical: N件, High: N件, Medium: N件, Low: N件
>
> レポート: `$OUTPUT_DIR/security-audit-YYYY-MM-DD.md`
>
> Critical/Highの項目について、今すぐ修正に進みますか？
> それとも一旦レポートを確認いただきますか？」

**修正を進める場合:**
- Critical → High → Medium の優先度順に対応
- 各修正ごとにコミット
- 修正完了後、レポートの該当項目に `[FIXED]` マークを付与

### Step 7: 保存とコミット

```
docs: add security audit report YYYY-MM-DD
```

修正を行った場合は修正内容ごとに個別コミット:

```
fix: sanitize user input to prevent SQL injection

Security audit: A03 injection - path/to/file.py:42
```

## 補足ルール

- **偽陽性の最小化**: 確信度の低い検出は `[要確認]` マークを付け、ユーザーに判断を委ねる
- **コンテキスト考慮**: テストコードやモック内の「脆弱性パターン」は意図的なものとして除外する
- **フレームワーク保護の考慮**: フレームワークが標準で提供する保護機能 (例: Rails の CSRF トークン) は、明示的に無効化されていない限り安全とみなす
- **修正の影響範囲**: 修正提案は既存の動作を壊さないことを前提とする。影響範囲が大きい場合はユーザーに相談する
- **定期実行の推奨**: 監査レポートに前回からの差分を含めることで、継続的なセキュリティ改善を促す
